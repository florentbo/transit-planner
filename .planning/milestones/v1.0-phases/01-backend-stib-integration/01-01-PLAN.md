  <action>
Create `StibWaitingTimesResponse.java` in `com.transit.client` package as Java records mapping the STIB OpenDataSoft API JSON response. The STIB API returns:

```json
{
  "total_count": 2,
  "results": [
    {
      "pointid": "8784",
      "lineid": "6",
      "passingtimes": "[{\"destination\": {\"fr\": \"ELISABETH\", \"nl\": \"ELISABETH\"}, \"expectedArrivalTime\": \"2024-12-27T18:51:00+01:00\", \"lineId\": \"6\"}]"
    }
  ]
}
```

CRITICAL: `passingtimes` is a JSON-encoded STRING, not an object. You need a custom Jackson deserializer that reads the string value and then parses it as a List of PassingTime objects. Use `@JsonDeserialize(using = PassingTimesDeserializer.class)`. Register `JavaTimeModule` in the inner ObjectMapper and set `ADJUST_DATES_TO_CONTEXT_TIME_ZONE` to false so OffsetDateTime keeps its original timezone offset.

Record structure:
- `StibWaitingTimesResponse(int totalCount, List<Result> results)` with `@JsonProperty("total_count")` and `@JsonProperty("results")`
- `Result(String pointId, String lineId, List<PassingTime> passingTimes)` with `@JsonProperty("pointid")`, `@JsonProperty("lineid")`, `@JsonProperty("passingtimes")`
- `PassingTime(Destination destination, OffsetDateTime expectedArrivalTime, String lineId)` with `@JsonProperty` annotations
- `Destination(String fr, String nl)`

Do NOT use Lombok (the current project does not use it). Use plain Java records.

Create `StibApiClient.java` as a Spring `@Component` with:
- Constructor takes `baseUrl` and `apiKey` (injected from config — use `@Value("${stib.api.base-url}")` and `@Value("${stib.api.key}")`)
- Method: `StibWaitingTimesResponse fetchWaitingTimes(List<String> pointIds)` that:
  - Builds URL: `{baseUrl}/catalog/datasets/waiting-time-rt-production/records?where=pointid={id1} or pointid={id2}&apikey={apiKey}`
  - Uses `java.net.http.HttpClient` with 5-second timeout (per user decision)
  - Parses response body with Jackson ObjectMapper into StibWaitingTimesResponse
  - Throws a RuntimeException (or custom StibApiException) on non-200 response or timeout

Add to `application.yml`:
```yaml
stib:
  api:
    base-url: https://stibmivb.opendatasoft.com/api/explore/v2.1
    key: ${STIB_API_KEY}
```

The `${STIB_API_KEY}` will fail at startup if the env var is not set (Spring behavior), which matches the user decision "fail at startup if STIB API key is missing."

Create `StibWaitingTimesResponseTest.java` as a JUnit 5 unit test that:
- Defines a JSON string matching the real STIB API response format (with passingtimes as embedded JSON string)
- Deserializes it with ObjectMapper
- Asserts: total_count, number of results, pointid, lineid, passingTimes list size, destination.fr, expectedArrivalTime value and timezone offset preserved

Use assertj assertions (already on classpath via spring-boot-starter-test).
